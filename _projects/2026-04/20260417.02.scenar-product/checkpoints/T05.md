## What was built

### Engine reconciliation (prerequisite)

Two drifts between the proto contract (T01) and the engine (T03) were resolved:

1. **Inline interactions**: `ScenarioStep<T>` now has `interactions?: readonly StepAction[]` inline. The old `StepInteractions = Record<number, readonly StepAction[]>` index-keyed map type was removed entirely. Both interaction hooks (`useBrowserStepInteractions`, `useTimeSourceStepInteractions`) read interactions from `steps[stepIndex]?.interactions` instead of a separate map.

2. **ActionType alignment**: Engine string literals now match proto enum names ‚Äî `"set_cursor"`, `"clear_cursor"`, `"scroll_to"`, `"viewport_transition"` (were kebab-case `"set-cursor"`, etc.). All matchers in hooks, effects, and warnings updated.

**Files changed in engine**: `types.ts`, `step-action.ts`, core barrel, react barrel, both interaction hooks, `warnings.ts`, `dom-helpers.ts`, `set-cursor.ts`, `viewport-transition.ts`, both READMEs.

### @scenar/sdk (new package)

A pure TypeScript authoring layer bridging the proto contract with the engine. Zero React dependency.

**7 source files, 472 LOC. 3 test files, 554 LOC. 31 tests.**

#### TS-first builder ‚Äî `createScenario()`

```ts
const scenario = createScenario({
  viewport: { width: 896, height: 540 },
  views: { settings: SettingsView, form: FormView },
  steps: [
    { view: "settings", delayMs: 0, props: { org: "acme" } },
    { view: "form", delayMs: 1500, props: { defaultName: "demo" },
      interactions: [{ atPercent: 0.2, type: "type", target: "name", text: "key" }] },
  ],
});
```

- `props` statically typed against the component registered under each `view`
- Framework-agnostic ‚Äî views are "anything that accepts props", not required to be React components
- Runtime validation: empty steps rejected, unknown views rejected with the registered list

#### Proto adapter ‚Äî `loadScenarioFromProto()`

- Accepts proto-generated `Scenario` messages via structural typing (no stubs dependency)
- Validates resource envelope (`apiVersion`, `kind`)
- Maps proto `ActionType` enum (numeric) ‚Üí engine ActionType strings (direct name copy since Phase 1 alignment)
- Expands proto `config` oneof ‚Üí flat optional fields on engine `StepAction`
- `InvalidScenarioError` with `.path` and `.reason` on any validation failure

#### Proto structural types

- `ProtoScenario`, `ProtoStep`, `ProtoStepAction`, etc. ‚Äî interfaces that structurally match protoc-gen-es v2 output without importing from generated stubs
- Consumers pass proto-generated objects directly; TypeScript structural typing validates compatibility

## Design decisions applied

1. **Proto is canonical**: Engine types were migrated to match proto shapes, not the reverse. See `design-decisions/006-t05-engine-interactions-inline-migration.md`.
2. **ActionType string alignment**: Engine uses proto enum names directly. See `design-decisions/007-t05-actiontype-snake-case-alignment.md`.
3. **SDK framework agnosticism**: Zero React imports. `ViewRegistry` accepts any callable. See `design-decisions/008-t05-sdk-framework-agnosticism.md`.
4. **Wrapper component deferred**: No `<Scenar>` wrapper ‚Äî deferred to post-T06 when real demo rewiring reveals the pattern. See `design-decisions/009-t05-scenar-wrapper-deferred.md`.
5. **Structural proto types**: SDK defines structural interfaces matching generated stubs instead of importing from them. Avoids coupling SDK to stub file paths.
6. **No Struct helper needed**: protoc-gen-es v2 auto-converts `google.protobuf.Struct` ‚Üí `JsonObject`. No separate `structToJson` call needed.
